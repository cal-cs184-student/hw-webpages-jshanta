Overview

In this assignment, I built a physically-based path tracer from scratch. Starting from ray generation through a virtual camera sensor, I implemented ray-triangle and ray-sphere intersection using the Möller–Trumbore algorithm and the quadratic formula respectively. I then built a Bounding Volume Hierarchy (BVH) to accelerate scene traversal from O(n) to O(log n) per ray, making complex scenes like maxplanck.dae renderable in seconds instead of hours.

Next, I implemented direct lighting via both uniform hemisphere sampling and importance sampling, followed by full global illumination using recursive path tracing with Russian Roulette termination. Finally, I added adaptive sampling to concentrate computation on difficult pixels and avoid wasting samples on already-converged regions.

The most interesting insight from this assignment was seeing how dramatically indirect lighting changes the visual quality of a scene — the color bleeding between walls and soft shadows under the bunny are entirely absent with only direct lighting. Russian Roulette also elegantly solves the infinite recursion problem in an unbiased way, which was a satisfying theoretical result to implement in practice.

Part 1: Ray Generation and Scene Intersection

Ray Generation and the Rendering Pipeline

Ray generation begins in Camera::generate_ray(x, y), which takes normalized image coordinates in \([0,1]^2\) and produces a ray in world space. The normalized coordinates are first mapped to the virtual sensor plane at \(Z = -1\) in camera space, where the sensor spans from \((-\tan(\frac{hFov}{2}), -\tan(\frac{vFov}{2}))\) to \((\tan(\frac{hFov}{2}), \tan(\frac{vFov}{2}))\). The mapping is: \[ \text{sensor}_x = (x - 0.5) \cdot 2 \cdot \tan\left(\frac{hFov}{2}\right), \quad \text{sensor}_y = (y - 0.5) \cdot 2 \cdot \tan\left(\frac{vFov}{2}\right) \] This gives a direction vector \((\text{sensor}_x, \text{sensor}_y, -1)\) in camera space, which is rotated into world space using the camera-to-world matrix c2w and normalized. The ray originates from pos with min_t = nClip and max_t = fClip.

In raytrace_pixel, for each pixel \((x, y)\) we generate ns_aa rays by jittering within the pixel using gridSampler, normalizing by image dimensions, and averaging the radiance returned by est_radiance_global_illumination across all samples.

Triangle Intersection: Möller–Trumbore Algorithm

The algorithm solves for where a ray hits the plane of the triangle and whether that hit point is inside the triangle. We express the hit point both as a point along the ray \(\mathbf{o} + t\mathbf{d}\) and as a barycentric combination of the triangle's vertices, yielding a 3×3 linear system solved efficiently using cross and dot products.

Concretely: we compute two edge vectors from \(p_1\), then use cross products with the ray direction to find barycentric coordinates \(u\) and \(v\). If either is negative, or if \(u + v > 1\), the hit is outside the triangle. Otherwise we compute \(t\) and check it falls within \([\text{min\_t}, \text{max\_t}]\). If all checks pass, we interpolate the vertex normals using barycentric weights \((1-u-v,\ u,\ v)\) to get a smooth surface normal.

Part 2: Bounding Volume Hierarchy

BVH Construction Algorithm

The BVH is built recursively. First, we compute the bounding box of all primitives in the current range and create a new node with that box. If the number of primitives is at or below max_leaf_size, we make it a leaf node and store the iterator range directly.

Otherwise we split using longest-axis centroid splitting: we look at the extent of the bounding box along each axis and split along whichever axis is longest. The split point is the centroid of the overall bounding box along that axis. We use std::partition to reorder primitives so those whose bounding box centroid falls below the split point go left, and the rest go right. If all primitives fall on one side (degenerate case), we force a median split to avoid infinite recursion, then recurse on the two halves.

The longest-axis heuristic works well because it splits along the most spread-out dimension, producing more balanced trees and tighter bounding boxes than always splitting along a fixed axis.

BVH Acceleration Analysis

Without BVH, rendering requires testing every ray against every primitive — O(n) per ray. With BVH, the tree allows the renderer to prune entire subtrees when a ray misses a bounding box, reducing tests to O(log n) per ray. In practice, cow.dae (~5856 triangles) that took tens of seconds without BVH renders in under a second with it. Scenes like maxplanck.dae (~50k triangles) are practically impossible without BVH but render in about one second with it. The speedup grows dramatically with scene complexity — the more primitives, the greater the benefit of spatial partitioning over brute-force intersection testing.

Part 3: Direct Illumination

Hemisphere Sampling Implementation

In estimate_direct_lighting_hemisphere, we sample uniformly random directions in the hemisphere around the hit point. For each sample direction, we cast a shadow ray and check if it hits a light source. If so, we accumulate the contribution using the rendering equation: \(f(\mathbf{w}_o, \mathbf{w}_i) \cdot L_e \cdot \cos\theta\), divided by the PDF \(\frac{1}{2\pi}\) (i.e., multiplied by \(2\pi\)). The final estimate is the average over all samples.

Importance Sampling Implementation

In estimate_direct_lighting_importance, instead of sampling random hemisphere directions, we sample directions directly toward each light source using sample_L. For each light sample, we cast a shadow ray with max_t = distToLight - EPS_F to check for occlusion. If unoccluded and the light is in front of the surface (\(\cos\theta > 0\)), we accumulate \(f \cdot L \cdot \cos\theta / \text{pdf}\). Point lights are sampled only once since all samples would be identical. The result is averaged per light.

Analysis: Hemisphere vs. Importance Sampling

Uniform hemisphere sampling and importance sampling both estimate the same direct lighting integral, but differ significantly in efficiency. Hemisphere sampling shoots rays in random directions and only counts a sample if it happens to hit a light — which is unlikely for small light sources, producing very noisy results even at high sample counts. Importance sampling instead samples directions directly toward each light source, meaning every sample has a chance of contributing useful information. This makes importance sampling dramatically more efficient: it converges to a clean image much faster and handles point lights correctly (which hemisphere sampling cannot, since the probability of a random ray hitting a point is zero). The tradeoff is slightly more complex implementation, but the noise reduction at equal sample counts makes importance sampling the clear practical choice.

Part 4: Global Illumination

Indirect Lighting Implementation

The function at_least_one_bounce_radiance recursively estimates multi-bounce lighting. At each hit point, we first add direct lighting via one_bounce_radiance (if accumulating bounces, or if we're at the first bounce). We then sample a new outgoing direction using sample_f, which returns both a direction w_in in object space and the associated PDF. We transform w_in to world space, cast a new ray from the hit point, and if it intersects the scene, recursively call at_least_one_bounce_radiance. The recursive contribution is weighted by \(f \cdot \cos\theta / (\text{pdf} \cdot p_{\text{continue}})\).

Russian Roulette termination uses a continuation probability of 0.65. The first indirect bounce always occurs (no RR applied); subsequent bounces are subject to random termination. This gives an unbiased estimate of the infinite sum of light bounces.

Part 5: Adaptive Sampling

Algorithm and Implementation

Adaptive sampling avoids using a fixed high number of samples per pixel by detecting convergence statistically and stopping early for pixels that have already converged. For each pixel, we track two running sums: \(s_1 = \sum x_k\) (sum of illuminances) and \(s_2 = \sum x_k^2\) (sum of squared illuminances), where \(x_k = \text{radiance}_k.\text{illum}()\). Every samplesPerBatch samples, we compute: \[ \mu = \frac{s_1}{n}, \quad \sigma^2 = \frac{1}{n-1}\left(s_2 - \frac{s_1^2}{n}\right), \quad I = 1.96 \cdot \frac{\sigma}{\sqrt{n}} \] If \(I \leq \texttt{maxTolerance} \cdot \mu\), the pixel has converged with 95% confidence and we stop sampling. Otherwise we continue up to ns_aa maximum samples. The actual number of samples taken is stored in sampleCountBuffer to generate the rate image.
